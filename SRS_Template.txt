1. Purpose
	This document outlines the requirements for Group 1's corporate communication application.
        1.1. Scope
			This Document will catalog the user, system, and hardware requirements for Group 1's corporate communication application. It will not document how those requirements are to be implemented.
        1.2. Definitions, Acronyms, Abbreviations
			1.2.1. Definitions:
				1.2.1.1. Client: The user-facing program for interacting with the system. Clients will connect to a singular server.
				1.2.1.2. Server: The program designed to host the system. Multiple clients connect to this server.
				1.2.1.3. Administrator: An elevated user with privileges that allow monitoring server chat logs, creating and deleting users.  
				1.2.1.4. Message: A single string of text and it's associated metadata
				1.2.1.5. Conversation: A series of messages between a set of users
				1.2.1.6. Read Message: A message which the system assumes has been read by a user
        1.3. References
			Use Case Specifications - Section 5
			UML Use Case Diagrams Section 6
			Class Diagrams – Section 7
			Sequence Diagrams – Section 8
        1.4. Overview
			Group 1's corporate communication application is designed to provide rapid wireless communication for all employees of a very large company. Messages sent will be immutable and records of every message sent 
			will be logged and accessible to internal IT users. The application will be interactable through a GUI, and connections will be serviced over TCP/IP. 
2. Overall Description
        2.1. Product Perspective
			 This product will be a highly scalable communication platform built for large organizations. It will facilitate text-based communication between a very large number of users via a client-server architecture. It will provide a simple GUI, and robust moderation and logging features. 
        2.2. Product Architecture
			The system will be organized into 3 major modules: the Server module, the Client module, and the GUI module
        2.3. Product Functionality/Features
			2.3.1. The high-level features of the system are as follows:
			2.3.2. Sending & receiving text through TCP/IP
			2.3.3. Text logs in the form of txt files
			2.3.4. User and admin login
			2.3.5. GUI for client side
			2.3.6. Separate programs for client and server
			2.3.7. Group and direct messaging
        2.4. Constraints
			2.4.1 All messages must be immutable once sent.
			2.4.2 No security features
			2.4.3 Cannot send data outside of plain text
			2.4.4 All functionality must be provided by the app itself
			2.4.5 The system can not use a database.
        2.5. Assumptions and Dependencies
			2.5.1 It is assumed that each client is connected to the internet to send and read messages
			2.5.2 It is assumed that users do not connect to a VPN during an app session
3. Specific Requirements
        3.1. Functional Requirements
            3.1.1. Common Requirements:
				3.1.1.1 All users will have registered accounts containing a username, password, real name, id, and position in the company. 
				3.1.1.2 The system will not allow users to delete their accounts
				3.1.1.3 There should be two categories of User: regular users and IT users
				3.1.1.4 All user accounts will be created either by IT users or hardcoded
				3.1.1.5 The system must allow users to send text messages to other individual users
				3.1.1.6 The system must allow users to create “group chats” between multiple users and send messages to those group chats.
				3.1.1.7 Messages must be immutable once they are sent.
				3.1.1.8 Messages sent must be within UTF-8 format.
				3.1.1.9 All conversations must be logged, including participants and other information about the conversation specific to group chats. 
				3.1.1.10 All individual messages must be logged, including the conversation it is a part of, text, and timestamp.
				3.1.1.11 The system should allow up to 20 messages per second per thread, and 
            3.1.2. Client Module Requirements:
				3.1.2.1. The Client application should not need to store any data locally except what is included in the application
				3.1.2.2. The client application should be able to connect to the server application via TCP/IP. 
				3.1.2.3. The application should connect to the server on opening, and not do anything else until the client is connected to the server.
				3.1.2.4. If the client is not connected to the server within 30 seconds, the client will continue to attempt to establish connection with the server, however the client will display a notification to the user indicating poor connection.
				3.1.2.5. The client application should request a set size array of message classes from a specific conversation as needed from the server based on user actions. 
				3.1.2.6. Regular users should be able to:
					3.1.2.6.1. Find any other registered users via name, username, or Id
					3.1.2.6.2. Start a conversation with and send messages to any other user
					3.1.2.6.3. Receive messages from any other user
					3.1.2.6.4. View all conversations had and the messages therein
					3.1.2.6.5. Create group chats and add other users to group chats
					3.1.2.6.6. Leave group chats
				3.1.2.7. IT Users should be able to do all of the above, and additionally
					3.1.2.7.1. View all conversations between all users and the messages therein
					3.1.2.7.2. Create new user accounts
					3.1.2.7.3. Edit login information for regular users
				3.1.2.8. The system should keep track of whether a user is logged in or not
				3.1.2.9. The system should keep track of whether a message has been read or not. 
            3.1.3. Server Module Requirements:
				3.1.3.1. The server should listen for message classes to be stored in a conversation class while it is running
				3.1.3.2. The server should store all data for users, conversations, and messages in plaintext files. 
				3.1.3.3. The server should create regular backups of this data
				3.1.3.4. The server should be able to connect to and provide data to any number of connected clients at a time. 
				3.1.3.5. The server should keep connections open until the client closes them.
				3.1.3.6. The server should be able to send data whenever the client requests it.
				3.1.3.7. The server should be able receive any data sent from the client, and send that data to other clients. 
				3.1.3.8. The server must ensure that ordering of data is maintained in logging and transmission.
            3.1.4. GUI Module Requirements:
				3.1.4.1. The GUI should open to the login screen, and not allow a user to do anything else until they login successfully
				3.1.4.2. The GUI should allow users to view all conversations they are a part of, and all messages in those conversations. 
				3.1.4.3. The GUI should display messages in chronological order based on timestamp.
				3.1.4.4. The GUI should provide a search bar to find other users
				3.1.4.5. The GUI must present input text from the user as the user is typing it.
				3.1.4.6. The GUI should allow users to see information about other users 
				3.1.4.7. The GUI should allow users to report messages and those reports should be sent to an IT User
				3.1.4.8. The GUI should detect when a conversation is being viewed by the user and consider all messages being viewed as read.
				3.1.4.9. The GUI should notify a user when they have unread messages in conversations they are not currently viewing.
        3.2. External Interface Requirements
			1.2.1. The system should not interface with any external systems
        3.3. Internal Interface Requirements
			3.3.1. The server should save all logs and user data in csv files it can recover from 
			3.3.2. Every conversation will be stored in an individual file, message data in that conversation will be stored in that file, conversation data will be stored in a header. 
			3.3.3. All user data will be stored in one file 
			3.3.4. All data shall be transmitted between the client and server over TCP/IP using UTF-8 encoding
			3.3.5. Data will be transmitted as a string of text which can be parsed into the proper objects
			3.3.6. The server will connect on Port number 54927
4. Non-Functional Requirements
		4.1. Security and Privacy Requirements
			4.1.1. No encryption of any data is to be used
			4.1.2. A message logs are to be readable by admins/IT Users
			4.1.3. All messages and logs thereof are immutable once sent
        4.2. Environmental Requirements
			4.2.1. The system (client, server, GUI) must be coded entirely in Java and written using the Eclipse IDE
			4.2.2. Use of any external libraries must be approved by the stakeholder 
			4.2.3 The system must be complete by 05/07/2026
        4.3. Performance Requirements
			4.3.1 The system should be scalable to be able to handle an arbitrary number of connected clients simultaneously
